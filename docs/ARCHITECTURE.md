# Архитектура (для Этапа 3)

## Принцип
**State-driven game**: всё поведение строго в рамках состояний, смена состояния = единственный механизм “перехода” (Title → Overworld → Battle → …).

## Слои
1) **Core**: цикл, ввод, стейт-машина.
2) **Gameplay**: логика мира/боя/прогрессии.
3) **Render**: функции рисования (без логики).
4) **UI**: HUD и экраны.

## Потоки данных
- Ввод → State.update() → изменение модели → State.render() использует render helpers.
- UI события (кнопки) не лезут в бизнес-логику: они вызывают методы state (например endTurn()).

## Почему так (для будущего Этапа 3)
- Можно усложнять бой и карту, не переписывая цикл игры.
- Можно заменить Canvas 2D на WebGL/движок, не меняя gameplay API.
